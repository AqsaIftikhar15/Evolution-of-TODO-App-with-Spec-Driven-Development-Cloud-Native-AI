# Todo In-Memory Python Console Application

**Phase I - Hackathon 2 | Governor Initiative for Artificial Intelligence & Computing (GIAIC)**

A command-line todo application demonstrating spec-driven, agentic development workflow using Claude Code and Spec-Kit Plus. This project showcases how to build software entirely through AI-assisted specification refinement without manual coding.

## AI Agents & Tools Used

This project was built using **Claude Code** with the **Spec-Kit Plus** framework, following a fully agentic development workflow:

### Development Workflow

```
Constitution → Specification → Planning → Task Breakdown → Implementation
```

### Agents/Commands Used

| Command | Purpose | Output |
|---------|---------|--------|
| `/sp.constitution` | Define project principles | `.specify/memory/constitution.md` |
| `/sp.specify` | Create feature specification | `specs/001-todo-cli/spec.md` |
| `/sp.clarify` | Review and clarify spec ambiguities | Validated spec completeness |
| `/sp.plan` | Generate implementation plan | `specs/001-todo-cli/plan.md` |
| `/sp.tasks` | Break down into executable tasks | `specs/001-todo-cli/tasks.md` |
| `/sp.implement` | Execute all tasks via Claude Code | Complete implementation |

### Key Principles

- **Spec-Driven Development**: All behavior originates from written specifications
- **Agentic Code Generation**: 100% code generated by Claude Code (zero manual coding)
- **Traceability**: Every implementation decision traceable to specification
- **In-Memory Only**: No persistence layer (Phase I constraint)
- **Console-First**: CLI interface only (Phase I scope)

## Features

### Phase I - Core Todo Management

- **Add Task**: Create tasks with title and optional description
- **View All Tasks**: Display all tasks with clear status indicators
- **Update Task**: Modify task title and/or description by ID
- **Delete Task**: Remove tasks by ID
- **Toggle Status**: Mark tasks as complete or incomplete

### Task Data Model

Each task contains:
- **ID**: Unique sequential integer (auto-generated, immutable)
- **Title**: Required text (user-provided, mutable)
- **Description**: Optional details (user-provided, mutable)
- **Status**: Complete or Incomplete (mutable, defaults to incomplete)

## Installation

### Prerequisites

- Python 3.13 or higher
- UV package manager ([Install UV](https://github.com/astral-sh/uv))

### Setup

```bash
# Clone the repository
git clone <your-repo-url>
cd "hackathon 2"

# Create virtual environment
uv venv

# Activate virtual environment (Windows)
.venv\Scripts\activate

# Or for Linux/macOS
source .venv/bin/activate
```

## Usage

### Run the Application

```bash
# Option 1: Using UV (recommended)
uv run python -m src.main

# Option 2: With activated virtual environment
python -m src.main
```

### Application Menu

```text
========================================
       TODO LIST MANAGER
========================================
  1. Add Task
  2. View All Tasks
  3. Update Task
  4. Delete Task
  5. Toggle Task Status
  6. Exit
========================================
```

### Example Workflow

```text
1. Choose option 1 to add a task
   - Enter title: "Buy groceries"
   - Enter description: "Milk, eggs, bread"

2. Choose option 2 to view all tasks
   - See your task with ID 1, status [ ] (incomplete)

3. Choose option 5 to mark task complete
   - Enter task ID: 1
   - Status changes to [x] (complete)

4. Choose option 6 to exit
```

## Development Philosophy

### Spec-Driven Development

This project demonstrates a **specification-first** approach where:

1. Requirements are documented before implementation
2. Specifications are refined through clarification
3. Implementation plans are generated from specs
4. Tasks are broken down systematically
5. Code is generated entirely by AI agents
6. All decisions are traceable to specifications

### Agentic Development

**Zero manual coding** - all code was generated by Claude Code following the spec-driven workflow. This approach:

- Validates the feasibility of AI-driven development
- Demonstrates spec quality through successful code generation
- Ensures complete traceability from requirements to implementation
- Enables rapid iteration through spec refinement

## Constraints & Design Decisions

### Phase I Constraints

- **In-Memory Only**: Tasks are lost when application exits (intentional)
- **No Persistence**: No files, databases, or external storage
- **Console Interface**: No GUI or web interface
- **Standard Library Only**: No external dependencies
- **Single User**: No authentication or multi-user support

### Rationale

These constraints focus Phase I on:
- Core functionality validation
- Spec-driven workflow demonstration
- Clean code architecture
- Foundation for Phase II expansion
